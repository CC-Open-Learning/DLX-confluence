Simulation Design

As a student, I need to see a realistic simulation of a transport truck and trailer, in a relevant industrial environment, so that I can learn to perform a pre-trip truck inspection

User sees realistic visual representation of a typical class 8 transport truck and trailer. The vehicle must show typical and realistic wear and tear, as well as minor and/or major faults, so that the learner can recognize and assess damages to equipment prior to operating the vehicle.

N/A no inspection interaction at this level.

Late model class 8 transport truck with cargo van trailer attached is shown. The truck must be presented on initial load in a typical “parked overnight” secured state.

Truck should be cold (not left idling), hood and all doors should be shut and properly secured. Trailer wheels (if trailer is present) should be chocked. The location should be an outdoor parking area, such as a truck stop, trucking yard, or parking lot.

TI-30 - Weather and environment To Do

As a student, I need to practice performing the pre-trip inspection in various types of weather, to reinforce the idea that the safety inspection always needs to be completed regardless of the weather conditions.

User experiences different realistic weather conditions (sunny, heavy rain, wind, maybe snow) to reinforce the idea that pre-trip inspections need to be completed in every kind of weather.

N/A no inspection interaction at this level

Weather conditions such as sunny, windy, heavy or light rain (possibly including thunderstorm effects) will be included in the background and particle effects.

In this release, weather will not actually affect the inspection process or outcomes at all. We will pretend that puddles of engine fluid will not be washed away in a heavy rain, even though we recognize that might happen in real life, depending on the conditions.

TI-31 - Guided Tutorial To Do

As a student, I need a Tutorial so that I can learn how to navigate and interact with the digital learning simulation.

Learner who is playing the sim for the first time will be launched into the Tutorial automatically.

Otherwise, learner clicks on the button in the main menu to start the Tutorial.

Tutorial must be displayed as the first item in the list.

Possibly, only the Tutorial is available the first time the learner accesses the simulation - the other modes do not become available until a) the Tutorial is completed, or b) the user exits the Tutorial.

Considering the scope of the tutorial, it may be worthwhile to have a sample “inspection” to give learners an example of what they will find in the main learning experience.

N/A - No inspection interaction at this level

Option A: Learner is presented with a menu containing buttons for all three of the “play modes”. Only the “Tutorial” option is interactable.

Option B: Learner is offered only one button in the UI “Start Tutorial”.

Subsequent launches will offer 3 options: Tutorial, Guided Mode, Advanced Mode

Tutorial will provide the learner with an orientation of the simulation controls: moving around, panning the camera, inspection UI, completing inspection tasks, difference between Guided and Advanced modes

TI-31 - Guided Tutorial To Do

Optionally Skip Tutorial

As a student I want to be able to skip the tutorial if I don’t need to learn about how to move around in a simulation, so that I can access the instructional portion of the simulation.

Learner clicks Exit in the toolbar to bring up a dialog prompting them to skip or continue the tutorial.

If the tutorial is continued, the menu is simply dismissed.

If the tutorial is skipped, the learner will then be presented with the mode selection menu

N/A no inspection interaction at this level

The “Dialog” window provided by CORE Engine can be manipulated at runtime to display the two buttons “Continue” and “Skip”, with the appropriate callback actions assigned to these buttons.

While learners must start the tutorial on their first time engaging with the simulation, learners who already know how to navigate around a simulation should be able to skip the tutorial. This also allows users to quit the tutorial if they accidentally select it as a menu option.

TI-52 - Select Play Mode To Do

Select Guided or Advanced Play Mode

On every subsequent launch of the simulation, the learner will have options in the main menu, prior to loading into the sim: Tutorial, Guided, Advanced.

As a student I want to see a main menu when I start the simulation so that I can select the play mode that I want to begin.

Learner is presented with a list of buttons for each of the “play modes” in the following order:

  • Tutorial Moved to main toolbar so that it remains accessible while the user is playing.

  • Guided/Instructional Mode

  • Advanced/Self-Assessment Mode

Selecting one of the buttons will begin that play mode.

The first time the learner accesses the simulation, the Tutorial should be the only play mode that that is available will start automatically when the user selects Guided or Advanced

The other modes do not become available until:
a) the Tutorial is completed, or
b) the user exits the Tutorial.

Descriptions for these play modes when they are disabled can instead direct the learner towards the Tutorial

On further consideration, the Tutorial has been moved inside the Guided or Advanced play mode so that it can be accessed from the toolbar at any time.

First time playing either Guided or Advanced mode, the Tutorial will load automatically will be offered in the toolbar with a reminder that the student can access it at any time.

TI-32 - Guided Mode To Do

As a student, I want to have the option to be guided through the inspection process so that I can learn how to conduct the inspection and learn what to look for.

User clicks to select “Guided” mode in the initial menu UI.

Guided must be the middle option in the list.

Learner will be guided through the inspection process with “on rails” navigation. At each step of the inspection, the voiceover will provide reminders about what to look for and/or rationale for why this step of the inspection is important.

Learner may not proceed to the next step of the inspection process until all of the required inspection interactions at the current step have been completed.

When the learner starts the sim in Guided Mode, they will not be required to navigate manually (no View menu or nav options). The simulation will “walk them through” the inspection in a logical pre-determined order.

To prevent this feeling too much like an instructional video that they just passively watch, we will maintain user interaction by having the learner decide WHEN to navigate, but not WHERE. The student is responsible to perform each step of the inspection, but does not have the opportunity to skip or forget any steps.

In Guided mode, we will lead the learner through each step of the inspection in a pre-set order, and we will provide immediate feedback at each step of the inspection. The voiceover can provide pointers for each step, calling the learner’s attention to key points.

TI-33 - Advanced Mode To Do

As a student, I want an Advanced play option that will not guide or prompt me through the inspection, so that I can gain confidence and practice my skills.

User clicks to select “Advanced” mode in the initial menu UI

Advanced must be the final option in the list.

Learner does NOT receive immediate feedback about missed steps in the inspection OR missed “bad” states. When the learner decides that they have completed their inspection, they will click to complete their inspection (e.g. Submit, I’m Done, etc) and only then will they find out if they missed major or minor damages.

In Advanced mode, the learner follows a less directed (optional alternate path) inspection process, but without the voice prompts or visual “help” identifying damaged components. Feedback will not be provided immediately, but will only be provided at the end of the inspection when the learner commits their inspection results.

This may include disaster scenarios when critical items are missed in the inspection.

TI-54 - Sounds To Do

Voiceover

As a student, I want to hear prompts, tips, and reminders while I’m playing so that I can get the support I need to successfully complete the simulation.

Voiceover will be most prevalent in Tutorial mode, where we will explain how to interact with the simulation and what the learner should expect.

Voiceover will also be a prominent feature in Guided Mode, where it can be used to remind the learner what to look for, or prompt them to try again if they have not completed their inspection tasks successfully.

Voiceover will be only occasionally heard in Advanced Mode, and only to provide feedback or correction when important errors are made.

TI-165 - UI To Do

As a student, I need to focus on reporting damage on my virtual inspection checklist so that I don’t spend a lot of my time recording results for components that are in ‘good’ condition.

Visual Inspection flow is when the inspection requires a visual assessment only (e.g. windows, mirrors, exterior damage). There is no physical interaction required for the learner to complete the inspection, they can complete the inspection just by looking.

If the learner sees damage on the model, they click on the model to report damage.

On Click - Inspection UI opens already in context for the selected component. There is no need for the learner to report if a component is “good”. The Inspection UI will only offer options for damaged/bad states.

When multiple components require inspection at the same POI (e.g brake lights, turn signal lights, emergency hazard lights, marker lights) the inspection UI will open on the specific component selected.

If the learner opens the Inspection UI but has no damage to report, they can close it using the X in the top right corner.

In guided mode, the learner will get instant feedback when they report damage (correctly identified damage will show in green, incorrectly identified damage will be shown in red)

In advanced mode, there is no immediate feedback for the learner. Their selection will be highlighted in blue, and they will not receive feedback until they have completed the inspection and submitted their results.

User clicks on specific damaged/bad component on the model to report damage.

There is no need for the user to click on the model if they do not see damage/bad state.

If the learner does not see any damage at this POI that they wish to record in their inspection results, they can move on to the next POI without reporting any damages/bad states. This could be something like a ‘Continue’ button or “Next Inspection” option.

In guided mode, the learner would be prompted to take a closer look if they try to move on without reporting damage that exists at this POI. In advanced mode, the learner would not be prompted - they do not get feedback until the end of the inspection.

Inspection UI is only shown if the learner clicks on the model to indicate that they see damage/bad state they wish to report.

TI-165 - UI To Do

As a student, I need to simulate inspection activities that would require interaction with the truck in the real world, without becoming confused about clicking only on damaged/bad components in the virtual inspection

Physical Inspection flow is when the component being inspected requires learner interaction in order to be evaluated, such as checking the engine oil with a dipstick, or checking tire pressure with a mallet.

User must trigger the start of the inspection activity - this could be a “Check the oil” button or similar control in the UI. The user should not click on the model to trigger the evaluation activity - they need to click in the UI or toolbar or on a button.

When triggered, the evaluation will be presented as an animated “cut scene” - such as seeing the dipstick being removed, wiped, replaced, and then the camera will zoom in to show the oil level on the dipstick. The learner will once again only click on the model if they have something to report.

If the learner clicks on the dipstick, the Inspection UI will open so that they can select the “bad state” that they want to report.

User clicks on the model after the cut scene, only if they want to report damage/bad state for the component.

If everything looks good, the learner will click the “continue” option to indicate that they have Next Inspection and they are ready to move on to the next POI.

Option to trigger the evaluation activity (e.g. remove the dipstick, strike the mallet on the tire, etc) is shown.

Inspection UI is only shown if the learner clicks on the model to indicate that they see damage/bad state to report.

TI-165 - UI To Do

As a student, I need to exit the simulation whenever I want to change to a different play mode or end my e-learning session, without being required to play all the way to the end.

The Exit button will be available and enabled in all play modes. The location of the Exit button should be consistent at all times, meaning that it should always be in the same position on the screen or in the UI in every play mode. This will make the button easy to find, and unlikely that the user would click it by accident.

The Exit button will have slightly different behaviour depending on where they are where it is clicked:

From the Tutorial: Confirmation dialog is shown because the user will lose their progress if they exit without completing the tutorial. If the user clicks to confirm they want to exit, they are returned to the mode selection menu.

From anywhere in the Guided or Advanced mode simulation: No confirmation dialog is shown. Progress is automatically saved, so the user does not require a warning before exiting. When the user clicks Exit, they are returned to the mode selection menu.

From the Mode Selection menu: Confirmation dialog is not shown. When user clicks Exit from this menu, they are returned to the Start menu.

N/A no inspection step

Exit button is available and enabled in all screens.

Feedback from users in the HE and MP simulations was that users found it tiresome to be reminded about losing their progress every time they clicked Exit. In those simulations, the confirmation dialog was required because all progress would be lost if the user exited. Now that we have Save & Load, we believe that the confirmation dialog is no longer necessary, so it has been removed from all but the Tutorial (we do not save progress in the tutorial)

Because the Exit button does not require confirmation, it is important that we arrange the toolbar so that accidental clicks on Exit are unlikely. For example, we would want to avoid placing a very frequently clicked button right beside the Exit button, to help reduce the chances that someone would exit accidentally. If this isn’t possible, we might need to consider moving the exit button to a separate location to help avoid accidental clicks.

TI-55 - Progress Checklist To Do

Progress Checklist

As a student, I need a quick view where I can see my progress through the inspection process, so that I can review which inspection steps have been completed and which are left to do.

User will trigger a read-only view of a progress checklist on demand (e.g. toolbar button, icon, hotkey)

The progress checklist will clearly show all of the inspectables that have been evaluated so far, and may optionally include an indicator for inspection tasks that have been started (partially completed) but not yet completed. The indicators must not be X’s or checkmarks, because those symbols are reserved as incorrect/correct indicators in the Results Panel.

The progress checklist does not provide any feedback about whether selected answers were correct or incorrect. That information will be provided via immediate feedback in the Inspection panel (in Guided Mode) or after the student has finalized their inspection and submitted it (Advanced Mode - via Results Panel).

The progress checklist is NOT interactable and cannot be used to record or edit inspection findings, nor to navigate. It is not clickable at all, and is a visual reference that summarizes learner progress and interim inspection results.

N/A - not inspectable

At any point in the inspection, the progress checklist can be displayed on demand and will reflect up to date progress through the required inspection tasks.

For clarity and ease of use, we will consider a “nesting” model for the checklist, where similar inspections are grouped together under a broader heading. Each heading could be designed to expand when clicked on, so that the student can obtain a more detailed breakdown if needed.

It will be tempting to model the checklist after the MTO inspection checklist, but it is important to note that an exact replica would be inappropriate. Our progress checklist should focus on what has been done and what needs to be done, as opposed to giving a detailed record of all the damages reported.

TI-388 - Results Panel To Do

As a student, I need a way to receive feedback about my selected answers in Advanced Mode, so that I can learn to correctly identify problems with my truck.

After “submitting” their final inspection results, the student will automatically be shown the Results Panel.

The results panel will list all of the inspectables (perhaps in the same order that was used in the Progress Checklist, if that makes items easier to find.

For each inspection POI, the student will receive feedback to help them evaluate their work:

Green checkmark - Item is completed, selected answer is correct.

Red X - Item is completed, selected answer is incorrect. Gives correct answer.

Incomplete - The student did not visit this POI. This step was forgotten or skipped.

The results panel is NOT interactable and cannot be used to change inspection findings. It is not clickable at all, and is a visual reference that summarizes learner progress and interim inspection results.

N/A - not inspectable

The Results Panel can only be viewed after the final inspection report is “submitted” in Advanced Mode.

The Results panel is not needed for Tutorial or Guided Modes.

TI-246 - Save and Load To Do

TI-248 - As a learner, I want to be able to resume playing where I left off, if I need to exit the scenario before completing all tasks. Done

Progress will automatically be saved anytime the learner exits either Guided or Advanced mode without completing all tasks. On subsequent launch, as soon as the learner selects the Play Mode, the simulation will check if a previously saved instance exists for that play mode for this learner. If a save file exists, the learner will be offered the option to resume where they left off, or restart from the beginning. The learner can have 0 or 1 save file for each of Guided and Advanced modes. No progress is saved for Tutorial mode.

N/A

The most recent progress will always be saved, overwriting any previously saved progress. Anytime the learner completes all tasks in a scenario without exiting, the save file will be discarded.

Save and Load will capture:

  • Current position (POI, view) of the learner at the time of exit

  • Previously selected inspection results (visible in progress indicator checklist)

  • “Seed” that was in use at the time of exit (dictates the states of all inspectable components on the model, so that the model will appear exactly as it did in the previous session)

Requires changes to the Task Manager system and Graph Controller to allow the simulation to load at a specific vertex rather than only at the beginning.
