VARLab : Digital Twin - Ohm's Law & DC Circuit Trainer Board

Created by David Inglis, last modified on Jan 09, 2024

Repository

https://bitbucket.org/VARLab/mpc-trainer-digital-twin/src/development/

Summary

The trainer board digital twin has been forked from the original Motive-Power-Circuits repository which contained the procedural labs 1-7.

MPC-Trainer-Digital-Twin is a sandbox-style environment which aims to provide a full-featured realistic simulation of the Ohm's Law & DC Circuit Trainer Board. The repository was forked in order to preserve the existing procedural labs while making the necessary changes to the underlying systems used by the trainer board.

Goals

  • A trainer board which allows for leads to be placed in ports

  • Functional digital multimeter for 20V DC ( and more? )

  • Components on the trainer board which simulate their real-world counterparts, ie bulbs, fuses, resistors, transistors, overload.

  • As a “digital twin” the virtual trainer board is expected to function exactly like a real-life trainer board (with some minor exaggerations in effects to make it easier for the learner to notice conditions such as blown fuses or circuit overload). This means that the required behaviour is for EVERY component on the virtual trainer board to work just like it does in real life.

  • Anything on the board that does not reliably work as it does in real life can be recorded as a bug and then prioritized according to its impact on the learning experience (just like we do with all of our bugs). In cooperation with the CEs and ID, we can determine which components are mission-critical for learner success and which are less important.

Risks

Risk

Mitigation Strategy

Validating circuit correctness is difficult for engineers without domain knowledge.

Documentation - future developers at VARLab must have access to required information about how to maintain and further develop the virtual twin

Additional components not included in current set of labs may require the additional dimensions of time in order to be accurately simulated.

Prioritization of any feature deficits will ensure that learners are not (or minimally) impacted by any components that do not behave as expected.

Back-porting changes from MPC-Trainer-Digital-Twin to Motive-Power-Circuits (Labs)

Separation - There is no known reason why the digital twin functionality would ever need to be backported to the procedural labs (Labs 1-5).

  • If someone did want to use the digital twin for Labs 1-5, they can! No further development would be required, this is an option.

  • The highly structured procedural nature of the Motive-Power-Circuits modules (already released as Milestone 1) does not require digital twin functionality.

  • Separating the digital twin into a dedicated repo prevents any unintended regression of the procedural labs.

Requirements

A single scene, the Circuit Sandbox, will be home to the trainer board digital twin and all functionality.

The scene will contain minimal UI; a settings menu which allows users to adjust audio and accessibility settings is required, however additional interfaces for guidance or feedback as in the original procedural labs will be omitted.

Timeline

In order to establish an estimation of project timeline and resources required, we must consider the following:

  • All of the tasks that a learner needs to be able to complete to finish labs 6-9 such as placing/stacking leads, adjusting knobs, taking measurements on the DMM, removing/replacing consumable components such as light bulbs and fuses.

  • New conditional behaviours such as O.L. light, fuses (incluuding DMM fuse) blowing, variable brightness of light bulbs

  • Flawed components such as blown fuses, burned out light bulbs, damaged cables - these are required for Labs 8 and 9. These must be visually identical to regular components, but will have additional resistance.

  • Number of development resources that we think we will need to deliver the required functionality

  • Approximate date (month?) when we think the required functionality will be ready to be tested by students

For purposes of estimation, we will assume that the digital twin design will stand alone:

  • No instructions, no user prompts, no feedback in-game

  • No knowledge checks

  • No voiceover

  • All lab procedures are outside of the DLX, probably in an accompanying document that a learner might have open in a separate app or browser tab.

Technical Process

SpiceSharp

The open-source cross-platform electronic circuit simulator library SpiceSharp is being leveraged to create the backend data model for the circuit simulation.

In the front end, many of the individual components on the circuit board will be reusable from the original labs, as well as the cable control system which allows for leads to be connected to ports on the trainer board and DMM.

Unity components will be data bound to their backend component representations, not unlike the process used in the LSM1.00 - Hydraulics simulation. This will map Unity objects onto the data model and keep them decoupled from the specific 3D model. That will also support testability and maintainability and ideally stability of the code base as well.

We need to validate that SpiceSharp has enough functionality to create this data model in code and once we've confirmed that then we can begin to create the trainer board sandbox.
