20 Aug 2024

First storyboarding meeting with Corey, Lisa, Nilupul, Sundar, and German

We still have some work to do in terms of marking the differences between Guided and Unguided workflows for the Procedural segments of the DLX, but we were able to identify a familiar workflow for the Equipment Orientation segment. Even though we know we won’t have all of the art assets that we need to finish this segment, we’ve decided to start our development effort with this segment because we are confident about how it should work. The general look and feel of the Equipment Orientation segment for RT will somewhat resemble the “Explore mode” workflow for AMB. Here are some notes about what we know so far:

ALL IN ONE MODULE - We previously thought that we would publish different segments of this DLX as individual WebGL modules, but Corey wants everything in one module with a big menu to allow students to browse through all available segments. Due to increased file size, this could result in slightly longer load times, but this is normal and we will do our best to optimize.

Equipment Orientation is only in Guided mode (there is no advanced version of this segment).

Setting: This can be done in an empty room for now.  We can find out later if the setup should take place in a supply area or at the patient bedside or a procedure room or what.

Camera/Navigation: First person POV, stationary camera.  No navigation, we only need pan & zoom for this segment. 

Character(s): No Patient or Physician NPCs are required for the equipment orientation. Mentor NPC can be voiceover at this stage, we can decide later if/how we want to represent that character in the scene.

  • There are no procedural steps (no tasks, no prompts, no feedback, no playtimer) required for this segment.

  • Progress is just an organized (expandable?) list of all items.  When the learner clicks on an item, that item gets a checkmark in the list. Segment is "complete" when all items have been clicked on at least once.  There is no limit to how many times a learner can explore the same POI or the same item, this is an unlimited segment.

  • Two main POI's for this scene are: Bronch Tower and Supplies.

  • For the Bronch Tower, the equipment will be front and centre in the scene, with hover highlight on the interactable parts and hover tooltips to display the name of the highlighted item.  

  • For the Supplies, we will use an expandable menu of some kind along with the Object Viewer.  Object Viewer is perfect for this interaction because it allows the learner to turn the 3D model over and look at the label as well as looking at the item itself. Every object in the menu will be labelled with its name, and will highlight on hover. When selected, the item will be launched in the Object Viewer to allow the learner to inspect it more closely, as well as displaying detailed information (name + other info, such as what it is used for). From a learning perspective, we chose NOT to put the supplies into the scene during the Orientation segment because we do not want students to accidentally memorize items by location (3rd drawer down on the left) instead of learning to recognize supplies using their label, appearance, and use. There are many options for how the supplies menu might be styled or laid out or organized, and we can decide that later.

  • The learner can click on each explorable item (part, button, port, whatever) to get detailed information (name + other info, such as what it is used for. A "listen" icon of some kind will be available so that the learner can hear the item name read by the voiceover.  We may optionally include another "listen" icon for the other information, in case we want voiceover for that too.  For now, the info can go into a dialog similar to the one used in AMB - we don't know exactly how this info will be styled in the finished scene yet, but we can decide later. The AMB dialog can be a placeholder while we are getting started.

  • Save & Load will save progress checklist only in the Equipment Orientation segment. 

Aug 7, 2024

Meeting with Stacey, Nilupul, Corey (Introducing Nilupul to the project)

  • At least 3 modules are expected:  Orientation, Guided Bronchoscopy Procedure, Advanced Bronchoscopy Procedure

    • Orientation: Recognize and know the use of the bronch tower equipment and supplies

    • Bronchoscopy Procedure modules: Scenarios and tasks are identical for both modules, but Guided has more learner supports enabled such as task prompts, task help, and enriched feedback.

  • Save & Load will be at the task level.

  • No need for navigation in this DLX, user does not move around the space.

  • Standard audio settings and camera controls are appropriate for this DLX

  • We may wish to include Guidance selector in the Settings menu so that a learner could choose more/less guidance, if multiple levels of guidance are included in the design

  • Storyboarding will begin with scenario definition - the focus will be on defining the scenarios (tasks,decisions) so that we can manage user pathing and make the scenarios as modular and reusable as possible, with appropriate variations to serve the learning objectives.

  • Visual elements of storyboarding should use VELCRO UI library as the basis for user interactions such as menus, feedback, just the basic UI elements that will be needed for all scenarios. There is no need to refine or 'design' the UI at this stage - just going for a general flow/layout type of representation.

  • Nilupul can review all of the procedural information and reference images/video in the shared workspace on SharePoint LSM9.05 Respiratory Therapy

  • Corey will add a copy of the RT project charter to that SharePoint workspace as well.

  • Procedural definitions should ideally be added to the SharePoint workspace, if comfortable sharing them.

  • Dev timeline is to create a Jira project (Stacey will arrange with David to ensure the most recent configuration is being used) and create a Unity project, including the integration of relevant packages.  Following internal storyboard review, we can get approval/feedback from the project stakeholders and start building the Jira backlog.  When there is enough Jira backlog ready for dev, we will recruit co-op developers and have formal dev kickoff.  Possibly as early as mid-September, but maybe not until October, depending on how quickly/well the storyboarding comes together
