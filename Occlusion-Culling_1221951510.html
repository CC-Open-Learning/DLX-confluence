What is Occlusion Culling?

Occlusion culling in its simplest description is derendering or hiding the models of objects (Occludees) not directly in view or that are being blocked by objects designated as an occluder.

Terminology

Static

  • This does not change during run time. These are things like walls and props that do not move. You can bake these because we know in the scene they will not be moving.

Occluder

  • an object that blocks visibility. This can be walls, or smaller objects like desks.

Occludee

  • an object that is hidden when out of view or behind an occluder.

Occlusion Area

  • A designated area where the player camera will be located, this is to help specify where more precise calculations should happen. If this is not created one is automatically made. Some unique issues may be solved by setting this up. I couldnâ€™t get this to make any significant differences so I did not implement this (I left it as the default automatic).

Occlusion Portal

  • Occlusion portal are components that you set up to dynamically block/unblock visibility at run time. This is good for things like doors because they need to block visibility when closed and not block when opened. Use this over static occluder flag for objects that are animated to move out of the way. I set this up using animation events and it worked pretty well.

Why use Occlusion culling?

If your scene contains many objects that will be in the background but might be hidden from the camera by other objects or walls it would increase performance to have those objects dynamically hidden. This is due to unity constantly trying to keep those objects rendered even though the user can not directly see it.

When not to use occlusion culling?

  • Objects will always be visible

  • Few amount of objects

Practical Results

When applied to Public Health Inspection DLX we saw a 40% increase in frame rate in the worst areas of the simulation. We cut down the amount of rendered triangle by almost half. This is a great result that increased the playability of the DLX significantly.

How To Implement Occlusion Culling?

First step to implement occlusion culling is to properly designate your objects as occluders and occludees. Doing this is simple, select your object from the unity hierarchy and in the inspection tab by the static drop down select whether you want your object to be and occluder, occludee or both.

If this prompts you to apply the changes to children I would recommend you do apply it unless there is a specific reason you do not want the children objects to have the same occlusion states.

After setting all the desired objects as occludees/occluders then set your occlusion setting and bake the occlusion data.

DO NOT UNDER ANY CIRCUMSTANCE SET DOORS AS STATIC OCCLUDERS. USE OCCLUSION PORTALS INSTEAD. Look below for more information

Occlusion Portal Setup

Occlusion Portal Setup (For Doors)

Occlusion portals are like occluder walls that you can turn on and off at run time. We use these because door need to let vision through but only when open.

To set them up add a occlusion portal component on your door object.

After that change the bounds of the portal to match your door, I prefer to overlap with the wall surrounding the door to ensure there are no gaps in the door way that would allow visibility lines through.

Finally set up your trigger to toggle the state of the portal as opened or closed. This can be done via script or animation events. I personally recommend animation events as they are easier to set up and handle.

Simply go the animation you wish toggle, add a reference to the occlusion portal open property and modify key frames for the start and end of every animation.

Baking And Settings

To access the occlusion window navigate to Windows â†’ Rendering â†’ Occlusion Culling.

This will open up an occlusion tab that you can set your desired settings.

Going from top to bottom the settings are:

  • Smallest occluder: This is the smallest size(in meters aka grid squares) that can block visibility lines. So objects smaller than this value will NOT block visibility. The lower this value the larger the occlusion data size will be and also lead to longer time for baking, nothing significant but good to keep in mind. I recommend a size of around 0.5-1. This is optimal because objects smaller than this likely wonâ€™t be really blocking visibility. Any bigger and you risk making objects that should be blocking visibility lines not work correctly.

  • Smallest Hole: This is the smallest sized hole your visibility should go through. Think of things like windows and gaps in fence grates. you want this to be a size that works for your scene. If you have small holes like in fences you wanna keep this number small. If you have only big holes like windows you can have this number slightly higher. Also, do not make the number too small because you might get issues with meshes having tiny holes that you donâ€™t notice making your occlusion culling worthless because you are still seeing through objects. I recommend 0.05-0.1

  • Backface Threshold: This reduces the detail by removing things like backface data. This is under the assumption that the camera often does not see the back of objects. This does have the potential to cause artifacts that cause issues. I recommend leaving this at 100 as there is more risk than benefit as I did not see a significant (or any) real changes when I tested this setting on Public Health Inspection.

After you have finished setting up occlusion setting click the bake button at the bottom right of the occlusion tab. This should bake your data.

I recommend clearing the data before re-baking anytime you make changes to the occlusion data. This is to avoid leftover data bleeding over into your new bake.

Visibility Tab

Visibility tab can cause issue with your scene when testing in the editor. Recommended to only briefly activate this tab to view the effects of culling and then immediately leave the tab as some things may appear as issues especially with using other cameras in the scene.

After baking you should be taken to the visibility tab if not you can switch to it by clicking on it to the right of the baking tab.

This allows you to see the scene with the effects of occlusion culling. You can select the player camera and enable visual lines to see what is blocking the cameras visibility and what the camera is rendering.

Notice how the wall and objects behind the camera are no longer visible? This is because they are being derendered.

Issues

Issues found with public health inspection

Visibility Lines Extending Past Occluders

There are 2 major issues I encountered while implementing occlusion culling. First was that unity renders slightly past objects that should block visibility. I am unsure if this is an issue with the thinness of the wall being uses, the modular design of our environment causing unity to get confused or simply how unity works. Basically objects that should have blocked visibility would have a slight distance they would render. Below is an example of what the issue is and how it affects our DLX in particular.

Overall, this wasnâ€™t an issue we were able to fix. If you wish to avoid this issue I would recommend designing your areas in a way that the occluder has some distance between it and the occludees or design your environments in a way that there are big enough gaps between walls for the visibility to be blocked before the next area begins.

Objects Disappearing When Viewed At An Angle

Other than this another issue encountered was with certain doors where looking at an angle that would move the door off-screen would cause it to derender while still in view.

Fixing this is easy simply remove the occludee static flag and the dynamic occlusion flag (donâ€™t forget to rebake!). This will ensure that the object will never be de-rendered like other objects.
