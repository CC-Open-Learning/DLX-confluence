In the shop safety portion, students will learn to identify hazards and inspect the state of the shop, including:

  • Hazards: like clutter, spills, and tripping hazards

  • Bench Grinder

  • Locations of Emergency and Fire Safety Equipment, including Hose Cabinet

  • Parts Washer · Tools Area · Overhead door

  • Floor Jack

In the hoist inspection, students will become familiar with the

  • Vertical Locks

  • Horizontal Locks

  • Hoist Arms

  • Hoist Arm Extensions & Pads

  • Height cut off bar

  • Equalizing cables

Objective

The objective of this module is to allow Automotive Service Technician student operators to inspect motive power equipment such as a hydraulic vehicle hoist in a virtual setting.

Daily Hoist Inspection Scenario

“Daily Hoist Inspection” Tasks

How will the inspection be implemented using tasks?

1. Raise the hoist to the 2nd set of vertical locks
The class that is used to raise the arm will implement ITaskable. For the “int” check type, it will return the last set of vertical locks that the arms passed.

2. Ensure that both sides engage on the vertical locks.
Only these 2 locks will be shown in the Inspection Checklist Window. The user will need to make the inspection on each lock.
This task will have 2 sub tasks, one for each lock. The task check will be to ask the Inspectable Manager if the component will be checked. The name of the component will be used to specify which component is being checked.

3. Ensure the hoist arms horizontal locks engage correctly.
Same process as 2

4. Manually operate the horizontal locks. Lubricate as necessary.
This task will have 4 sub tasks, one for each lock. The check type will me “Moved”. The taskables will need a new action “OnTaskableActive” that will fire when the TaskVertexManager starts checking the component. The locks will have a bool Moved variable that is set true whenever the lock is moved. The OnTaskableActive event will set Moved = false. This will let us track that each of the locks have been moved after the task started.

5. Show all the components in the Inspection Checklist Window. Complete the task once all the tasks are complete.

Post-Inspection Demo Scenarios

Below is the proposed design for demonstrations of a vehicle being lifted by the hoist (and the potential problems that arise), to be presented to the student after they have completed their vehicle hoist inspection.

After a design discussion outlining pros and cons of multiple solutions (captured in the table below), Approach 1 was selected as the current direction for development.

Proposed Solutions

Approach

Pros

Cons

1

Hoist inspection has a random set of defects. After inspection, the student is presented with a dialog allowing them to from a list of “demo” scenarios, including a "happy" path scenario and a set of "unhappy" path demos (not stating what the possible result will be)

  • more flexible in capturing the breadth of knowledge to be presented to student

  • allows for new “unhappy” demos to be added without compounding possible work for the student in the initial inspection

  • easier for developers to build, more modular

  • usable as a teaching tool, professor can present to class

  • less realistic, student's inspection doesn't have any bearing on demos

  • some confusion in the process from inspection into demo phase

  • takes the student out of the context of the lab when presenting dialog with demo scenarios

2

Hoist inspection has exactly one broken component, leads to exactly one unhappy path if missed. Otherwise, the student is presented with the "happy" demo

  • student sees the consequences of their actions

  • natural flow from inspection to post-inspection demo

  • randomization may lead to uneven distribution of demo scenarios

  • some students would likely choose not to do the "explore mode" if they don't have to

  • would have to repeat the initial inspection multiple times to get to all of the demo scenarios

  • if inspection is fully completed, demo is just a vehicle lift and lower with no problems. doesn't showcase possible bad outcomes

  • waste of dev time to create "unhappy" scenarios if the ideal outcome from inspection is to never see them

  • in developing this option, all backend functionality of approach 1 would still need to be developed

Implementation

After inspection, the student is presented with a dialog allowing them to choose from a list of “demo” scenarios. One of these demos will be a vehicle hoist with no issues; this will be the “happy” path scenario. The list will also contain “unhappy” path scenarios, each of which will relate to a potential defect of the vehicle hoist. The “unhappy” demos can be separated into two categories: “unhappy” paths arising from missed inspection tasks, and all other available “unhappy” demos.

The implemented post inspection window

In our implementation of the post inspection window, we decided to create the above. It features two columns: a green column which indicates the flaws with the hoist the student found, and a red column that indicates the flaws with the hoist that the student missed. The student can then click on each of the buttons in each column to view a simulation around what could happen had they missed the issue in real life. At the bottom of the window there is the “Fully Functional Hoist” button which allows the student to view a simulation of a the hoist when properly functioning.

Demo scenarios will have the vehicle positioned on the hoist arms, ready to be lifted. The student isn't responsible for positioning arms underneath the vehicle, and the only control they will have during the demos is to lift and lower the hoist using the hydraulic controls.

Bench Grinder

The user must ensure the grinder is in good working order and is free of damages.

  1. Ensure the lens is clear and unobstructed
    The learner will need to record whether the lens is clear or obstructed depending on the appearance. The obstructed lens can be dirty and unclear.

  2. Ensure the grinding wheel is in good condition
    The learner will need to record whether the grinding wheel is in good condition (not cracked) depending on the appearance. A broken grinding wheel can have a visible crack in it.

  3. Ensure Tool Rest is in good condition
    The learner will need to record whether the tool rest is in good condition depending on the appearance. Defective tool rest can be misplaced (ex. not at a right angle to the stone wheel).

  4. Ensure Bench Grinder Power Cable is in good condition
    The learner will need to record whether the cable is in good condition (not frayed) depending on the appearance. The frayed cable will have exposed wiring.

  5. Step out of the way of a bench grinder for safety
    Have a separate POI which will move the user to the side of the grinder before turning the machine on.

  6. Turn on the Bench grinder
    The learner will need to interact with the power button to turn the machine on.

  7. Ensure Bench Grinder operates correctly
    The learner will need to record whether the machine turned on properly or not. A defective bench grinder can have visible excessive vibration while on.

Hazards

Learning Objective

Hazards include cluttered workbenches, chemical spills, and tripping hazards. The learner must be able to identify where hazards may occur in the auto shop. The learner must also be aware of the problem/danger associated with the hazard.

Functionality

This section details what the hazard system should do. For information about how the hazard system is implemented, refer to the Implementation section below.

On mouseover a Hazard will be highlighted, some hazards may require custom highlighting functionality as described in MP-317 - Getting issue details... STATUS and MP-280 - Getting issue details... STATUS . The learner will indicate that they have seen a hazard by clicking on it. When a Hazard is clicked it will display a MessagePrompt at the top of the screen. When the prompt is clicked, the hazard is removed and the prompt is hidden.

At the end of the scenario, the learner is told the number of hazards that they’ve missed and forced to continue exploring the auto shop to find the missed hazards, or quit after entering explore mode.

Implementation

Hazards need to be randomly generated but do not need to be inspectable and therefore do not need to use the Inspection System. A system should be created to randomly activate interactable hazards using the Interaction System, which will be called a HazardController.

The HazardController was created with MP-289 - Getting issue details... STATUS and refined in MP-308 - Getting issue details... STATUS .

Since the HazardController works by activating Hazard objects/prefabs, all Hazard objects/prefabs should be disabled at the start of the scenario.

The Hazard class uses an enum (called HazardType) to determine the type of hazard. An additional HazardTypeInformation class is used to store the minimum/maximum hazard counts for a HazardType. Minimum/maximum hazard count refers to the number of hazards that can appear in the scenario for a HazardType. The HazardController has an array of HazardTypeInformation that provides default information about hazards, however, the defaults can be overridden in the Unity Inspector.

Progress on implementation of this system can be tracked from here: MP-288 - Getting issue details... STATUS .

Frequency of Hazards

The number of hazards that occur and the placement of hazards is randomized. For example, if there are five tripping hazards in the scenario, the hazard system will activate between one and five tripping hazards in the scenario. The locations of the hazards is also randomized. However, there is always one and only one chemical spill hazard in the scenario
