Reference will be made to the hydraulic labs module, and how design principles are applied.

Example of a simulation breakdown:

From Lab to Simulation

Thoroughly review the lab, take note of…

  • Theoretical applications (the logic of connected components and reactive states)

  • Physical objects (for 3D modelling, moving parts for users to interact with, theoretical constraints, etc.)

  • Emphasis of learning (ex. observation in audible cues from a Hydraulic Power Unit upon increasing its relative output pressure)

  • Steps and process (for user guidance, task verification, etc.)

Modelling of Real World to Virtual World

The 2D or 3D model should offer itself as a functional component, for the emphasis of learning within the user interactions of the simulations.

Find the answers to these questions…

  • What part(s) of the object are essential for developers to implement function?
  • What part(s) of it are essential for users to utilize its function?

Determining Value and Purpose

Ask…

Why?

  • What value does this bring for the user?

  • Does it align with the goal of the application?

  • Does it fit within the context of the application?

You can also apply some principles from root cause analysis, to help determine the purpose of implementing a certain feature.

Learn more about The Five Whys and how it helps boil down problems to the most fundamental principle.

Human Centered Interaction Design

The interaction should be based within the core actions required by the lab. UI elements should be minimal, and provide value without being a distraction.

Some of these points are an interpretation from the design principles of Don Norman.

Constraints

The limitations put in place for the user to interact with the interface.

Put yourself in the shoes of the person performing the activities in this lab.

Try to constrain the focus to the key area of where the users sustained attention is required.

For example...

In a hydraulics lab - students will be typically be working on a station with the parts available in a drawer. So, we keep our camera focused on this zone.

Visibility

Use universal symbols for iconography.

Utilize visual cues to form instinctive mental models to the objects in the simulation.

For example...

Building a menu? Try a Unicode character: ☰

The “hamburger” menu icon is universal, and you know when you click it something will expand.

Of course, if you are feeling creative - feel free to try and create a more appealing design.

Feedback

For every action, there should be some sort of response from the system.

For example...

For example, turning the Flow Control Valve (a knob per se) within a circuit will increase or decrease the pressure to the HPU. When this is done correctly, the HPU reacts with audible feedback, and other parts may react in a similar fashion. For example, the speed of retraction and extension of a cylinder.

What if the circuit is not connected properly, and the Flow Control Valve gives no response? Typically, in the real world the situation could be messy - but in the confines of a simulation we should try to give some sort of feedback for every action. Hence, an error message to let the user know that something is wrong. If it is their fault, or the systems fault.

Consistency

A particular action should always cause the same reaction.

For example...

This expansion box has a consistent reaction and purpose throughout the page.

Affordances & Mapping

In the hydraulics lab, we highlight clickable objects.

Having a fairly simple UI, the user may be inclined to click on such an object to see what happens.

From there, the user learns the function from the visual feedback provided to them.

Design enough to bring awareness of the object, and pique their curiosity to interact with it.
