Objective

Developers working on the Trucking Inspection Digital Learning Simulation should reference this development style guide to inform:

  • how interactable objects can best be set up in a scene

  • how inspectable objects can best be set up in a scene

  • how text can be displayed to the user

  • how windows and other interactable UI elements can be displayed to the user

With this guide, content created for the DLS will be consistent and give developers a point of reference when beginning their work, while still allowing for individual design and creativity as a team member working on this DLS project.

Vehicle Inspection

Inspectable Components

A multi-point vehicle inspection can be facilitated using the Inspection system. The Trucking Inspection simulation will be broken into segments where the learner will have a small number of objects to inspect in each segment.

Trucking will also use two distinct inspection modes: Visual and Physical.

The Design Considerations - CORE and Trucking document in the Trucking Inspection workspace outlines this design decision in full.

Learner Guidance

Throughout the inspection, voice clips will be played to guide the learner through the different inspection points, and will be used as hints or suggestions when it seems that the learner is stuck on a particular inspection point. Voice clips will primarily be used in the Tutorial and Guided modes to teach and guide the learner through the inspection points. In Advanced mode, voice clips can be used to kick off the inspection, and guide the user back to the inspection if they have idled for a while, but otherwise guidance should be left to a minimum.

Developers can use the Audio Controller in the scene to store all voice clips in a shared location as key-value pairs. The keys can then can then be used to easily reference a voice clip in the Scenario graphs any of the play modes.

Script: AudioController.cs

Voice clips can be accompanied by text displayed on-screen using a Message Prompt effectively as subtitles for the voice clip. This can be useful as an accessibility consideration to learners and faculty, and can help learners in the simulation if their sound is low or off, or they have missed part of the audio-based prompt.

Interactable Components

In the Trucking sim, learners will not be clicking and dragging to “interact” with objects manually, and the sim will not use the Interactable system in the same way that simulations like Motive Power or Heavy Equipment use it. Instead, learners will either be looking at an object for a visual inspection and only needing to interact with it to make note of damages, or they will be triggering an animation by selecting an option in the “Physical Inspection Button Tray” for a physical inspection.

For more information on using Interactables, review the documentation for the Interaction System.

User Interfaces

Inspections

For both the visual and physical inspections, learners will record damages that they notice by clicking on damage. Clicking on any inspectable area of the vehicle, whether it is damaged or not, will bring up an Inspection Panel with the possible damaged states. This is a modified version of the panel used in Heavy Equipment, but doesn’t give the option to select “good” as a state, since all components are assumed to be good unless some damage/defect is selected.

Displaying Text On Screen

Displaying text for the user can be done using the PromptManager system.

The TaskManager uses the PromptManager system to automatically display a text prompt at the bottom of the screen containing the description of the current Task. This automatic prompt can be disabled, or can be updated to show additional information to the learner such as hints.

Additional messages can be placed at the top or the middle of the screen:

  • Top-of-screen positioning is useful when providing additional context or information for the user’s task or action, since it will not be placed in front of whatever the learner is currently engaging with.

  • Middle-of-screen positioning should be used when the message is intended to interrupt the learner’s workflow; this can be used to provide a warning or cautionary message to the learner if they have performed an incorrect action.

The documentation on Message Prompts provides descriptions and code examples for setting up such prompts.

Displaying Windows and Tooltips

In addition to the tooltips/floating control panels of the inspection system, we might want to display a window to the user.

If the learner should be able to open and close this window/menu at their discretion, it should be added to the menu bar which is positioned at the bottom-left of the screen. An icon that is relevant to the purpose of the window will make the option clear to the user. An on-hover tooltip containing the name of the window should also be added to the button in the menu bar.

An example of this kind of window is the Magnifiers window which is used in both the Hydraulics and Motive Power (Daily Hoist Inspection) simulations. The learner can open the window to view any magnifiers at any time by clicking on the magnifying glass icon in the menu bar. They can then click the button again or click the “X” on the window itself to close the window; clicking on another button in the menu bar will also close the window, if it has been set up with the Window Manager to be a “sibling” window

A window can also be made to be entirely controlled by CORE Engine. It is then displayed to the learner at certain times as defined by the Scenario, and typically the learner must complete the action prompted by the window in order to close it. An example of this behaviour is seen in the Daily Hoist Inspection sim=

Any window that can be manually activated by the user can also be controlled by CORE Engine through the Scenario. Take the “Quit Dialog” for example: the learner can press the ESC key or click on the “back” arrow in the menu bar to “pause” the simulation and bring up the dialog allowing them to quit. This dialog is also automatically displayed to the learner once they have completed all of the task of the simulation.

See the RemoteEducation.UI documentation for more information about implementing windows, buttons, and tooltips.
