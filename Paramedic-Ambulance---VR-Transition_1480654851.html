Project Introduction

The Paramedic Ambulance is being transitioned to enable use of the Meta Quest 3 headset inside the simulation. This was intended to show off the simulation in another form for trade shows and demos, and was originally not intended to be a one-to-one mapping of the original project (despite a one-to-one mapping being later achieved).

System Changes Post-Transition

There were no large changes to the AMB system during the VR transition. The main changes that affected the system are:

  • Platform Controller

    • Listens for the Switch Platform key-bind and activates/deactivates the flat screen/virtual reality UI panels

  • CustomControllerManager

    • Extends and replaces the ActionBasedControllerManager to add additional controller action bindings

    • Allowed the use of the grip to call HandleCloseCompartment() to provide an alternative to hitting the return button manually

  • XR System

    • This was added to control all of the XR functionality and contains the controller logic, as well as a copy of all UI panels that were migrated to the XR Panel object type

      • XR Panels (XRMenu UnityObject) are enabled upon the Switch Platform button being hit, and the flatscreen UI panels are disabled, or vice-versa.

Simulation Setup to Play & Controls

Setup

To play the simulation on the Meta Queset 3, follow these steps:

  1. Install and setup Meta Quest Link

    1. Ensure that it is set to allow unofficial XR programs in the settings menu (in the Meta Quest Link app)

  2. Power on the Meta Quest 3 and connect to the Meta Quest Link app

    1. This can be done with a USB Type-C cable or through Wi-Fi

  3. Launch the Quest Link system on the Meta Quest 3

  4. Navigate to the Monitor that Unity appears on inside of the Quest Link desktop

  5. Press play on the simulation

  6. Press 'P' to switch the platform

    1. This will make the headset go black and should load the AMB-VR simulation directly onto the headset

  7. Play!

Controls

Left Controller:

  • Free Movement: Joystick

Right Controller:

  • Interaction (Select): Trigger

  • Return (Out of Compartment): Grip

  • Look Around: Joystick Left and Right

  • Teleport: Joystick Up

Keyboard:

  • Switch Platform: P

    • Switches between flat screen and virtual reality systems

Insights and Lessons Learned

Core gameplay functionality transitions with little to no effort. This is made possible by the use of the Core XR package and by the original functional design. AMB was designed to function the same no matter the method of play (flatscreen or XR) and transitions accordingly. Very little changes were made asides from adding controllers, copying UI elements to the Flex XR panel, and slight modifications to the Outlinable to allow the controller raycasts to highlight it.

Flatscreen UI does not maintain a good user experience through its use of clickable panels. The flow of play inside XR differs too greatly from flatscreen. Using the floating UI elements to navigate and interact (such as the Return button when in a compartment, or the X button on the info panel) feels sloppy and can be difficult to interact with. Use the controller buttons/triggers/grips for play feel much smoother and allow greater fluidity.

Also, flatscreen UI panels and the physical on screen buttons are useful for synchronizing the user’s actions with animations inside the system. However, free movement inside of the sim breaks this synchronization by allowing the user to walk out of the patient bay and into the cab interior while compartments are still open in the patient bay, as one example. This can then feel sloppy and confusing when the user is trying to interact with new compartments and return from them, but other compartments are still open and the controls are affecting systems outside of the user’s view.

One example of how flatscreen UI elements can be repurposed to better indicate necessary actions from the user to continue with new interactions, is to use a cascading indicator to show how ‘deep’ inside of a compartment you are. This shows how many times the user should/needs to return for them to continue. Due to lack of camera restrictions that force the user to use the return button, this may be necessary.

Another insight that was gained from demo feedback was the potential use of transitioning the checklist UI panel to be something that the user can toggle on and off and view from the left controller. This would allow it to function much like a notepad or clipboard that the user can ‘pull from their pocket’ to quickly and easily view their progress. This would free up more space as the checklist UI dominates the screen inside of VR and blocks field of view. This may also give the opportunity for users to rely more on their knowledge of the general compartments and items they can discover in explore mode, with the checklist acting as an aid, rather than guiding them strictly.

One last insight is that flatscreen play is much more universally accessible. I had family members try the AMB-VR sim, and there were some common issues of it not being very usable depending on eyesight/glasses, the headset feeling heavy and giving them a headache, or potentially causing neck-pain. I also experienced these and found VR disorienting and incovenient.
