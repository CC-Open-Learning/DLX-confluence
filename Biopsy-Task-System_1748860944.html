VARLab : Biopsy Task System

Created by User e88ce, last modified on Apr 15, 2025

The Biopsy Procedure module of this DLX is composed of several “Scenarios“, each representing a structured part of the overall module.

Each Scenario is composed of one or many “Tasks” defining specific interactions or instructional elements. These tasks are implemented as scriptable objects that inherit from the BiopsyBaseTaskSO class.

These are currently six TaskSO types:

  • PromptTaskSO

  • MessageTaskSO

  • ItemHandlerTaskSO

  • ItemRequestTaskSO

  • POIHandlerTaskSO

  • BronchMonitorTaskSO

Each TaskSO inheritor has its own implementation of Execute() which carries out the required code for that task. For more information on Scenarios, see HERE

To create any of the following:

Right-click > Create > RT Scriptable Objects > Biopsy > Scenario Tasks

--Base Class--

BiopsyBaseTaskSO

  • The abstract class all other Task classes inherit from

  • Class Properties

    • TaskIndexIncrement (bool) - a task will proceed to the next one after it is completed

    • ScenarioIndexIncrement (bool) - a task will increment to the next Scenario when it is completed

  • Methods

    • Execute() - executes the given Task, each Task class has its own implementation specific to what needs to occur

--Inheritors--

PromptTaskSO

  • Main purpose of this task is to display a PromptSO on screen to instruct/inform the user of their next action

    • a new PromptSO should be created as a property of this task

  • Class Properties

    • Name

    • PromptSO - includes the prompt information

      • Speaker - name of the speaker (i.e. Physician, Preceptor)

      • Message - description text of the prompt

      • Image - NPC image relating to speaker

      • Colour - Bar (left side bar), Speaker (highlight of speaker name), Image Border (highlight around circular image)

  • Execute() - calls prompt.HandleDisplayUI()

    • shows the prompt - dismissed after delay

MessageTaskSO

  • Main purpose of this task is to display a ModalSO on screen to instruct/inform the user of their next action

    • a new ModalSO should be created as a property of this task

  • Class Properties

    • Name

    • ModalSO - includes the modal information

      • Name, title, description, sub-description, note (blue text), button text, image, note image

  • Execute() - calls modal.HandleDisplayUI()

    • shows the modal - dismissed by button click

POIHandlerTaskSO

  • Main purpose of this task is to set the current POI, and open the current POI of interest

  • Class Properties

    • ProcedurePOI

      • selected from a POIType enum option

  • Execute() - calls poiManager.HandleExecutePOITask()

    • opens to the parameter POI (ProcedurePOI)

ItemHandlerTaskSO

  • Main purpose of this task is to enable a particular item in scene as item of interest

  • Class Properties

    • ProcedureEquipment

      • selected from a BiopsyProcedureEquipment enum option

    • EnableAllTableItems

      • bool - toggle option to set all procedure items to active in the scene

  • Execute() - calls biopsyProcedureComponentHandler()

    • enables the parameter BiopsyProcedureEquipment (ProcedureEquipment)

    • example - in the current scenario the Bronchoscope is the selected ProcedureEquipment and is active in the scene. Another task may require the Bronchoscope with Saline for example to be active in the physician’s hand

ItemRequestTaskSO

  • Main purpose of this task is to set an item on the procedure table as the RequestedItem. Certain Scenarios/Tasks require the user to identify a correct item amongst many, they are informed by prompt when they select a right/wrong item

  • Class Properties

    • RequestedItem

      • this is the item the user is needing to find on the table

        • it is a BiopsyProcedureEquipment enum option

    • RequestPrompt

      • the prompt that is displayed when the user is searching for the item

        • the same prompt as the PromptTask for the corresponding Scenario

          • when the user selects an incorrect item, a prompt is shown informing them of this, the RequestPrompt is displayed afterwards so the user knows which item they are searching for

    • Actions after finding the requested item

      • GoToNextTask - increments to next scenario task

      • GoToNextScenario - increments to next scenario

  • Execute() - calls tableController.SetRequestedItem()

    • sets the parameter RequestedItem as the item to be “checked”

    • when the user clicks on table items they will be informed if right/wrong based off this item

    • example - in this task, the Saline20cc is the RequestedItem, clicking other items triggers the WrongItemPrompt, RequestPrompt is displayed after a small delay

BronchMonitorTaskSO

  • Main purpose of this task is to display a set of images that correspond to actions the user is taking with regards to the Bronch Monitor. One scenario for example requires the user to advance and retract the forceps. As the user interacts with the UI, the BronchMonitorStatus is updated, and the corresponding image is displayed on the Bronch Monitor based off this status

  • Class Properties

    • BronchMonitorStatus

      • the current status of the screen (i.e. off, retract needle, advance needle, etc.)

        • this is a BronchMonitorStatus enum option

    • BronchMonitorDataList <List>

      • list of images that will be cycled through for a given task to reflect the current status of the monitor

    • Actions after finding the requested item

      • GoToNextTask - increments to next scenario task

      • GoToNextScenario - increments to next scenario

  • Execute() - calls bronchMonitorController.SetBronchMonitorData()

    • depending on the BronchMonitorStatus, displays the corresponding image, as the user completes the sub parts of the task, the BronchMonitorStatus changes, as does the current BronchMonitorDataList image being displayed
