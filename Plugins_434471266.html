VARLab : Plugins

Created by Colm Ambler (Unlicensed), last modified on Dec 15, 2021

This document will serve as both an introduction as to how plugins are implemented in games, and the plugins we’re using work.

Each plugin is implemented in projects very differently, it all depends on the developer. However, we integrate the files into our projects the same way.

Step

Reference

Description

1. Find the plugin folder

Within the project folder, there will be a js folder. Inside of that is a folder titled plugins. Each project is created with this folder, which will already contain files within. These default plugins are not integrated into projects, however.

2. Drag the plugin json file in the folder

Simply drag your plugin’s json file into this folder.

3. Open the plugin manager and bring the plugin into your project

After opening the plugin manager, simply click on one of the spaces and you will be taken to a settings tab. By clicking on the name, you’ll see that the json files are all in the project. Double click and that is all. Next, you’ll need to actually implement it. Each plugin handles their implementation differently.

Galv’s Layer Graphic

Link: https://galvs-scripts.com/2020/08/24/mz-layer-graphics/

Developer: Galv

Release Date: August 24, 2020

Version Used: 1.2 for MZ

Description: It allows us to use a single image as a parallax background with multiple options we can tweak. This allows us to have more specific designs for maps rather than being confined to a grid.

Use: We’re using it in three maps. The “A Building CSI Services” map due to the unique floor patterns. The bus to make the design simpler and the credits.

Implementation:

Step

Reference

Description

1. Open the img folder inside project file and create a folder called “layers”

The plugin will recognize the folder when you refer to it in the engine.

2. Drop the image you want to use inside

The name of the image doesn’t matter. We’re using the one that’s highlighted.

3. Inside the desired map’s properties, call upon the layer folder in notes and write out the specifications.

Inside the notes, you’ll input the specific file you want and the specifications. As reference, here is the line as currently in the game.

LAYER_S 0 AWINGFLOOR.V3 0 0 255 0.1 10 0 0

LAYER_S - This refers to the id of the map we are creating the layer from

0 - The id of the layer itself

AWINGFLOOR.V3 - The name of the image in the layers folder

0 - The speed of the x movement

0 - The speed of the y movement

255 - The opacity of the layer

0.1 - The Z level of the layer. 0 = ground, 5 = above all characters.

10 - The horizontal shift that differs from player movement.

0 - The horizontal shift that differs from player movement.

0 - Blend mode, 0 = normal.

There are other options as detailed in the plugin description, but they aren’t needed for our purposes.

PKD_SimpleQuestSystem

Link: https://kdworkshop.net/plugins/simple-quests-system/

Developer: Pheonix KageDesu

Release date: V. 1.0 June 10, 2021 Latest version (as of right now) 1.2.1 November 11, 2021

Version Used: 1.1 for MZ

Description: The quest log acts similarly to other quest logs seen in most modern RPGs. It can track active quests and completed quest. You can manage how far the player is in a quest and change the task and description accordingly. You can separate quests into different categories as well as create pointers for the player to follow to their destination.

Use: As of right now, we have 17 quests within the game. 5 for the main wayfinding quest, 8 for the wayfinding quest, 3 for side quests and 1 for the final scene of the game. We keep track and update the player on every action they’ve taken in the quest and update both the current task and description. We don’t use the pointers as this is a game about finding your way through campus and literally pointing in the correct direction defeats the purpose a bit too much.

Important Note: As noted above, we are not using the most updated version of the plugin. This is due to a bug that crashes the game in relation to the quest log being updated before having been opened. We attempted to circumvent this by forcing open the quest log whenever the player got a new quest (which still happens). It didn’t fix it though as a battle counted as a new scene and when the data was updated, it immediately broke. A developer at the VARLab, Jordan Robinson, was able to jerry-rig a patch so that it works. We’ve shown the error to the developer and they were already aware of it. As far as we know, it hasn’t been fixed which is why we haven’t updated to 1.2.1

Implementation: All of this info is available in the plugin description within RPG Maker but just for posterity, there’ll be a brief overview of it here.

Step

Reference

Description

  1. Make A quest

To create a quest, simply open the quests tab (it’ll be the first option in the plguin parameters) and then click on an empty space. In the case of this picture, an already existing quest is in the spot (at the bottom of the list is an empty quest, once that’s made a new empty entry is added and so on).

2. Add Tasks

Inside the quest is another set of parameters including quest name, ID, name for the in-game list, tasks, descriptions, etc. To add a task, click on task and add them similarly to how you added a quest. The first task is automatically shown when the quest is added to the game.

3. Add Descriptions

Descriptions are pretty much the exact same as tasks. The first one is automatically shown when the quest is started.

4. Create the logic in-game

Inside of an event, you’ll need to use the “script” option to edit the quests. There are multiple script-calls but the ones you’ll need to remember are “SQSM,AddQuest” “SQSM.ShowTaskForQuest(quest ID, task#)” “SQSM.ShowDescriptionForQuest(quest ID, task#)” “SQSM.CompletQuest”. Just to clarify, it is CompletQuest, not CompleteQuest. As mentioned above, the first quest task and description are automatic. In this picture, it calls the first task for the quest, but that’s a mistake, it would show up regardless.

5. In Game Reference

This should look similar to most other quest logs you see in video games. The plugin author specifically cites Skyrim as an inspiration. This picture is in the “Current Quests” tab, indicated by the book images in the top left. The name in the top bar and the list on the left are different instances of the same name and can be different if you so wish. The description and tasks are visible (in this case the first four tasks are all visible at once) and this is a list of all the quests. They can be divided into Main and Side quests.

VisuStella MZ Plugins

Link: https://visustella.itch.io/visumz-sample

Developer: Visustella

Release Date: August 20, 2020 (For all of them)

Version Used: 1.48 for MZ (Core Engine), 1.48 for MZ (BattleCore), 1.13 for MZ (MainMenuCore)

Description: VisuStella has a collection of free plugins that change many aspects of an RPG Maker project. These are all found by downloading the sample project.

Use: In the Campus RPG we are currently using the CoreEngine (which is necessary for all other VisuStella plugins to work), the BattleCore and the MainMenuCore. This document won’t contain all the information on each of the plugins because they simply change too much that it wouldn’t be practical to write it all here. Instead, there will just be some general information about why each of the three plugins are being used in the Campus RPG.

Core Engine: As said before, this plugin is needed for all other VisuStella plugins to work. This particular plugin acts as a sort of “unofficial patch” that fixes many of RPG Maker MZ’s bugs. This also allows us some customization in terms of the UI, screen motion, battle systems, keyboard inputs among others. As of right now, none of the settings for the CoreEngine have been edited.

BattleCore: This plugin completely overwrites and edits the battle system of RPG Maker both visually and mechanically. The most important part of this plugin to us is the addition of enemy health bars. This allows the player to feel that they’re consistently making progress in the encounters. This is especially important to those who have not played a video game as it might seem broken or as if they’re doing something wrong. This plugin also changes the colours of the numbers that appear on-screen depending on if it’s a player character or an enemy and if they’re losing EN or gaining EN. It also counts turns and gives the player an option for auto battling. Also Important to note that due to this plugin overwriting the basic RPG Maker MZ code there doesn’t seem to be a reliable way to turn off the MP gauge. The only method we found didn’t work so instead we just changed the offset so it wouldn’t be visible anymore. This was done in the BattleCore parameters.

MainMenuCore: This plugin allows for different pause menu styles and changes more elements of the UI. It shows playtime and portraits. There isn’t much else, it hasn’t been edited.
